## 模式一：用 AsyncGenerator 传递流式数据

**问题：** AI 模型是流式输出的，如何把 token 从 API 传到 UI，中间不积压？

**解法：** 用 `async function*` 贯穿整个调用栈。

```typescript
// 每一层都用 yield* 透传，不缓冲
async function* apiLayer() {
  for await (const token of llmAPI.stream()) {
    yield token  // 立刻传出去
  }
}

async function* businessLayer() {
  yield* apiLayer()  // 透传，不缓冲
}

async function* uiLayer() {
  for await (const token of businessLayer()) {
    renderToken(token)  // 拿到就渲染
  }
}
```

**好处：**
- token 从 API 直接流到 UI，延迟最低
- 消费者慢了，生产者自动等待（背压）
- 代码线性易读，错误处理简单

## 模式二：对话循环的标准结构

**问题：** AI Agent 的核心循环应该怎么写？

**解法：** while 循环 + 工具调用检测 + 最大轮次限制。

```typescript
async function* agentLoop(initialMessages, maxTurns = 10) {
  let messages = initialMessages
  let turnCount = 0

  while (turnCount < maxTurns) {
    turnCount++
    const toolCalls = []

    // 调用模型，收集工具调用
    for await (const event of callModel(messages)) {
      yield event  // 实时传给 UI
      if (event.type === 'tool_use') {
        toolCalls.push(event)
      }
    }

    // 没有工具调用，任务完成
    if (toolCalls.length === 0) break

    // 并行执行所有工具
    const results = await Promise.all(toolCalls.map(executeToolCall))

    // 把结果加入历史，继续循环
    messages = [...messages, ...results]
  }

  if (turnCount >= maxTurns) {
    yield { type: 'max_turns_reached' }
  }
}
```

**关键点：**
- 工具并行执行（`Promise.all`），不要串行
- 必须有最大轮次限制，防止无限循环
- 超出限制时明确告知用户

## 模式三：四层上下文压缩

**问题：** 对话历史越来越长，超出模型上下文窗口怎么办？

**解法：** 分层处理，从轻到重。

```typescript
class ContextManager {
  async prepare(messages, model) {
    // 第一层：删除中间冗余消息
    messages = this.snip(messages)

    // 第二层：压缩超长工具结果
    messages = await this.microcompact(messages)

    // 第三层：接近上限时，AI 生成整体摘要
    if (this.isNearLimit(messages, model)) {
      messages = await this.autoCompact(messages)
    }

    return messages
  }

  // 第四层：定期提取关键信息，跨会话持久化
  async extractMemory(messages) {
    const memory = await this.summarizeKeyInfo(messages)
    await this.saveToFile(memory)
  }
}
```

**关键点：**
- 每层都有电路断路器，失败时不无限重试
- 跨会话记忆用文件持久化，不依赖内存

## 模式四：电路断路器

**问题：** 某个操作反复失败，如何防止无限重试？

**解法：** 记录连续失败次数，达到上限就停止。

```typescript
class CircuitBreaker {
  private consecutiveFailures = 0
  private readonly maxFailures: number

  constructor(maxFailures = 3) {
    this.maxFailures = maxFailures
  }

  async execute<T>(operation: () => Promise<T>): Promise<T | null> {
    // 已经熔断，直接返回
    if (this.consecutiveFailures >= this.maxFailures) {
      console.log('Circuit breaker open, skipping operation')
      return null
    }

    try {
      const result = await operation()
      this.consecutiveFailures = 0  // 成功，重置计数
      return result
    } catch (error) {
      this.consecutiveFailures++
      if (this.consecutiveFailures >= this.maxFailures) {
        console.log(`Circuit breaker tripped after ${this.consecutiveFailures} failures`)
      }
      throw error
    }
  }
}

// 使用
const breaker = new CircuitBreaker(3)
const result = await breaker.execute(() => autoCompact(messages))
```

**适用场景：** 任何可能反复失败的操作：上下文压缩、API 调用、工具执行。

## 模式五：分层权限决策

**问题：** 工具调用的权限检查，如何在安全和效率之间平衡？

**解法：** 从快到慢的决策链，能快速决定的不走慢速路径。

```typescript
async function checkPermission(tool, input, context) {
  // 1. 拒绝规则（最快）
  if (matchesDenyRule(tool, input, context)) {
    return { allow: false, reason: 'deny rule' }
  }

  // 2. 允许规则（快）
  if (matchesAllowRule(tool, input, context)) {
    return { allow: true, reason: 'allow rule' }
  }

  // 3. 安全工具白名单（快）
  if (isSafeTool(tool.name)) {
    return { allow: true, reason: 'safe tool' }
  }

  // 4. AI 分类器（慢，但智能）
  if (context.mode === 'auto') {
    const result = await aiClassifier.classify(tool, input, context.history)
    return { allow: !result.shouldBlock, reason: result.reason }
  }

  // 5. 询问用户（最慢，但最准确）
  return { allow: await askUser(tool, input) }
}
```

## 模式六：子 Agent 默认完全隔离

**问题：** 多个 Agent 同时运行，如何防止互相干扰？

**解法：** 创建子 Agent 时，默认克隆所有可变状态，no-op 所有写操作。

```typescript
function createSubagentContext(parent, options = {}) {
  return {
    // 克隆，不共享
    fileCache: parent.fileCache.clone(),
    replacementState: parent.replacementState?.clone(),

    // 独立的 abort 控制器
    abortController: new AbortController(),

    // 默认 no-op，防止子 Agent 修改父 Agent 状态
    updateUI: options.shareUI ? parent.updateUI : () => {},
    updateState: options.shareState ? parent.updateState : () => {},

    // 独立的错误追踪
    denialTracking: createFreshDenialTracking(),
  }
}

// 使用：默认隔离
const subCtx = createSubagentContext(parentCtx)

// 需要共享时显式声明
const interactiveSubCtx = createSubagentContext(parentCtx, {
  shareUI: true,    // 共享 UI 更新
  shareState: true, // 共享状态
})
```

## 模式七：Prompt Cache 优化

**问题：** 多个 Agent 调用 API，如何减少重复计算，降低费用？

**解法：** 保持 API 请求前缀一致，命中 Prompt Cache。

```typescript
// 关键：子 Agent 使用与父 Agent 完全相同的系统提示和历史
type CacheSafeParams = {
  systemPrompt: string      // 必须与父 Agent 完全相同
  conversationHistory: Message[]  // 父 Agent 的完整历史
}

async function runSubagent(params: CacheSafeParams, newPrompt: string) {
  // 请求结构：[系统提示] + [父 Agent 历史] + [新任务]
  // 前两部分与父 Agent 完全相同 → 命中缓存
  // 只有最后的新任务需要重新处理
  const messages = [
    ...params.conversationHistory,
    { role: 'user', content: newPrompt }
  ]

  return callAPI({
    systemPrompt: params.systemPrompt,  // 相同
    messages,  // 前缀相同，命中缓存
  })
}
```

**效果：** 子 Agent 的 API 费用降低约 90%（缓存命中费用是正常费用的 1/10）。

## 模式八：根据 Agent 职责裁剪上下文

**问题：** 不同类型的 Agent 需要不同的上下文，如何避免浪费 token？

**解法：** 根据 Agent 的职责，有选择地注入上下文。

```typescript
async function buildAgentContext(agentType, baseContext) {
  // 只读 Agent 不需要 git 状态（可能有几千行）
  if (agentType === 'Explore' || agentType === 'Plan') {
    const { gitStatus, ...contextWithoutGit } = baseContext
    return contextWithoutGit
  }

  // 只读 Agent 不需要项目指令文件
  if (agentDefinition.omitProjectInstructions) {
    const { projectInstructions, ...contextWithoutInstructions } = baseContext
    return contextWithoutInstructions
  }

  return baseContext
}
```

**原则：** 每个 Agent 只看它需要的信息，不要一刀切地注入所有内容。

## 模式九：连续拒绝追踪

**问题：** AI 分类器一直拒绝某个操作，Agent 会陷入死循环怎么办？

**解法：** 追踪连续拒绝次数，达到上限时降级到人工确认。

```typescript
class DenialTracker {
  private consecutiveDenials = 0
  private totalDenials = 0
  private readonly limits = { consecutive: 3, total: 10 }

  recordDenial() {
    this.consecutiveDenials++
    this.totalDenials++
  }

  recordSuccess() {
    this.consecutiveDenials = 0  // 成功时重置连续计数
  }

  shouldFallbackToHuman(): boolean {
    return (
      this.consecutiveDenials >= this.limits.consecutive ||
      this.totalDenials >= this.limits.total
    )
  }
}

// 在权限检查中使用
async function checkPermission(tool, input, tracker) {
  const aiDecision = await aiClassifier.classify(tool, input)

  if (aiDecision.shouldBlock) {
    tracker.recordDenial()

    // 连续拒绝太多次，让人来决定
    if (tracker.shouldFallbackToHuman()) {
      return await askHuman(tool, input)
    }

    return { allow: false }
  }

  tracker.recordSuccess()
  return { allow: true }
}
```

## 模式十：跨会话记忆

**问题：** 每次新会话都从零开始，如何让 Agent 记住之前的工作？

**解法：** 后台定期提取关键信息，写入文件，下次会话时注入。

```typescript
// 后台记忆提取（不阻塞主对话）
async function extractMemoryInBackground(messages, memoryFile) {
  // 用独立的 Agent 提取，不影响主对话
  const summary = await runIsolatedAgent({
    prompt: `
      分析以下对话，提取关键信息：
      - 当前任务和进度
      - 重要发现和决策
      - 待完成的工作
      - 项目特有的约定和规则

      对话内容：${messages}
    `,
    tools: [onlyAllowEditFile(memoryFile)],  // 只允许编辑记忆文件
  })

  await writeFile(memoryFile, summary)
}

// 新会话时注入记忆
async function buildSystemContext(memoryFile) {
  const memory = await readFile(memoryFile)
  return {
    previousContext: memory,
    currentDate: new Date().toISOString(),
  }
}
```

**触发时机：** 不要每次都提取（太频繁），也不要太少（信息丢失）。Claude Code 的策略是：token 数增长超过阈值 AND 工具调用次数超过阈值，两个条件同时满足才触发。

## 总结：构建 AI Agent 的核心原则

从 Claude Code 的源码中，可以提炼出几个核心原则：

**1. 流式优先**
用 AsyncGenerator 传递数据，不要等所有数据都准备好再处理。

**2. 防御性设计**
每个可能失败的操作都要有电路断路器，防止无限循环。

**3. 默认隔离**
子 Agent 默认完全隔离，需要共享时显式声明。

**4. 分层决策**
从快到慢，能快速决定的不走慢速路径。

**5. 按需裁剪**
不同的 Agent 只看它需要的信息，不要浪费 token。

**6. 持久化记忆**
重要信息写入文件，不要只存在内存里。
