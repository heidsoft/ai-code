# 第六篇：AI Agent 怎么管理权限？

## ——从"每次询问"到"AI 自动判断"

---

Claude Code 可以执行 shell 命令、修改文件、访问网络。

这些操作都有风险。一个不小心，可能删掉重要文件，或者执行了危险命令。

Claude Code 是怎么控制这些风险的？

---

## 五种权限模式

Claude Code 有五种权限模式，从最严格到最宽松：

**1. default（默认模式）**
每次工具调用都询问用户是否允许。最安全，但最烦人。

**2. plan（计划模式）**
AI 只能制定计划，不能执行任何操作。用户审核计划后，手动批准执行。

**3. acceptEdits（接受编辑模式）**
文件编辑类操作自动允许，其他操作还是询问。适合信任 AI 改代码，但不信任它执行命令的场景。

**4. auto（自动模式）**
用 AI 分类器判断操作是否安全，安全的自动允许，不安全的询问用户。这是最智能的模式。

**5. bypassPermissions（跳过权限模式）**
完全跳过所有权限检查。只在完全隔离的沙箱环境中使用，非常危险。

---

## 权限决策链：从快到慢

每次工具调用，都会经过一个决策链。越靠前的检查越快，越靠后的越慢：

```
工具调用请求
  │
  ▼
[1] 检查 deny rules（拒绝规则）
    → 匹配到 → 直接拒绝
  │
  ▼
[2] 检查 allow rules（允许规则）
    → 匹配到 → 直接允许
  │
  ▼
[3] bypassPermissions 模式？
    → 是 → 直接允许
  │
  ▼
[4] acceptEdits 模式 + 文件编辑操作？
    → 是 → 直接允许
  │
  ▼
[5] 安全工具白名单？
    → 在白名单里 → 直接允许（跳过 AI 分类器）
  │
  ▼
[6] auto 模式？
    → 是 → 调用 AI 分类器判断
  │
  ▼
[7] 询问用户
```

**为什么要这样分层？**

因为每一层的成本不同：
- 规则匹配：几微秒
- 白名单查找：几微秒
- AI 分类器：几百毫秒（需要调用 API）
- 询问用户：几秒到几分钟

能用快速方法解决的，就不用慢速方法。

## 规则系统：精细控制

用户可以配置允许规则和拒绝规则：

```json
// ~/.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(git:*)",        // 允许所有 git 命令
      "Bash(npm run:*)",    // 允许所有 npm run 命令
      "FileEdit"            // 允许所有文件编辑
    ],
    "deny": [
      "Bash(rm -rf:*)",     // 禁止 rm -rf
      "Bash(sudo:*)"        // 禁止 sudo
    ]
  }
}
```

规则支持通配符，可以精细控制到具体命令前缀。

## AI 分类器：最有趣的部分

auto 模式下，Claude Code 用一个 AI 来判断另一个 AI 的操作是否安全。

这个分类器叫 **YOLO Classifier**（Yes Or Let's Observe，或者说"要不要放行"）。

```typescript
// src/utils/permissions/yoloClassifier.ts（简化）
async function classifyYoloAction(
  conversationHistory,  // 完整对话历史
  action,               // 要执行的操作
  tools,
  permissionContext,
  signal,
) {
  // 构建分类器的提示词
  const prompt = `
你是一个安全分类器。根据对话历史和用户意图，判断以下操作是否安全：

操作：${action}

对话历史：${conversationHistory}

判断标准：
- 操作是否符合用户的明确意图？
- 操作是否有不可逆的破坏性风险？
- 操作是否超出了任务范围？

请回答：允许 或 拒绝，并说明理由。
  `

  const result = await callSmallModel(prompt)  // 用小模型，速度快成本低
  return {
    shouldBlock: result.decision === '拒绝',
    reason: result.reason,
  }
}
```

**分类器的快速路径**

为了减少不必要的 AI 调用，有几个快速路径：

```typescript
// 1. 如果 acceptEdits 模式会允许，直接允许（不调用分类器）
const acceptEditsResult = await tool.checkPermissions(input, {
  mode: 'acceptEdits'
})
if (acceptEditsResult.behavior === 'allow') {
  return { behavior: 'allow' }  // 快速路径
}

// 2. 如果工具在安全白名单里，直接允许
if (isAutoModeAllowlistedTool(tool.name)) {
  return { behavior: 'allow' }  // 快速路径
}

// 3. 否则，调用 AI 分类器
const result = await classifyYoloAction(...)
```

## 连续拒绝追踪：防止 Agent 卡死

有一个容易被忽视的问题：如果 AI 分类器一直拒绝某个操作，Agent 会怎样？

它可能陷入死循环：
1. Agent 尝试执行操作
2. 分类器拒绝
3. Agent 换个方式再试
4. 分类器再次拒绝
5. 循环往复...

Claude Code 用**连续拒绝追踪**来解决这个问题：

```typescript
// src/utils/permissions/denialTracking.ts
export const DENIAL_LIMITS = {
  consecutiveDenials: 3,   // 连续拒绝 3 次
  totalDenials: 10,        // 总拒绝 10 次
}

// 连续拒绝达到上限时，降级到询问用户
if (shouldFallbackToPrompting(denialState)) {
  // 不再自动拒绝，而是弹出对话框让用户决定
  return { behavior: 'ask' }
}
```

逻辑是：如果 AI 分类器连续拒绝了 3 次，说明这个操作可能确实需要人工判断，自动拒绝已经不够了，应该让用户来决定。

## 子 Agent 的权限继承

Claude Code 支持多个 Agent 协作。子 Agent 的权限是怎么处理的？

```typescript
// src/tools/AgentTool/runAgent.ts（简化）
// 子 Agent 的权限上下文
const agentGetAppState = () => {
  const state = parentContext.getAppState()

  // 规则：父 Agent 的权限模式优先
  // 如果父 Agent 是 bypassPermissions，子 Agent 也是
  // 如果父 Agent 是 acceptEdits，子 Agent 也是
  if (
    state.toolPermissionContext.mode === 'bypassPermissions' ||
    state.toolPermissionContext.mode === 'acceptEdits'
  ) {
    return state  // 继承父 Agent 的宽松权限
  }

  // 否则，使用子 Agent 自己定义的权限模式
  if (agentDefinition.permissionMode) {
    return {
      ...state,
      toolPermissionContext: {
        ...state.toolPermissionContext,
        mode: agentDefinition.permissionMode,
      }
    }
  }

  return state
}
```

**异步子 Agent 的特殊处理：**

后台运行的子 Agent 无法弹出对话框询问用户（用户可能不在看终端）。所以异步子 Agent 默认设置 `shouldAvoidPermissionPrompts: true`，遇到需要询问的操作时，直接拒绝而不是等待用户响应。

## BashTool 的安全防御

BashTool 是最危险的工具，可以执行任意 shell 命令。它有 2593 行专门的安全防御代码。

主要防御层次：

**1. 命令语义分析**

```typescript
// 解析命令的 AST（抽象语法树）
// 识别危险模式
const dangerousPatterns = [
  /rm\s+-rf\s+\//, // rm -rf /
  /dd\s+if=.*of=\/dev/, // dd 写入设备
  /mkfs/, // 格式化文件系统
  // ... 更多危险模式
]
```

**2. 路径验证**

```typescript
// 确保操作不超出允许的目录
function validatePath(path, allowedDirs) {
  const resolved = resolve(path)
  return allowedDirs.some(dir => resolved.startsWith(dir))
}
```

**3. 输出重定向检测**

```typescript
// 检测命令是否有输出重定向（可能覆盖重要文件）
const { redirections } = extractOutputRedirections(command)
if (redirections.length > 0) {
  // 需要额外确认
}
```

**4. 沙箱模式**

在支持的环境下，可以在沙箱里执行命令，即使命令有破坏性，也不会影响真实系统。

---

## 小结

Claude Code 的权限系统体现了一个核心思想：**在安全和效率之间找平衡**。

- 明显安全的操作：快速自动允许
- 明显危险的操作：快速自动拒绝
- 不确定的操作：用 AI 判断，或者询问用户
- AI 判断失效时：降级到人工确认

这个分层设计让 Claude Code 在大多数情况下不需要打扰用户，同时在真正危险的操作上保持谨慎。
