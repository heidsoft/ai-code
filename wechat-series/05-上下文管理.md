# 第五篇：当对话太长怎么办？

## ——Claude Code 的四层上下文压缩策略

---

AI 模型有一个硬限制：**上下文窗口**。

就像人的短期记忆有限，模型每次能"看到"的内容也有上限。Claude 3.5 Sonnet 的上下文窗口是 20 万 token，大约相当于一本 200 页的书。

听起来很多？但在实际编程任务中，很快就会用完：
- 读了几十个文件
- 执行了几十次命令
- 来回对话了几十轮

怎么办？Claude Code 有四道防线。

---

## 先理解 Token 是什么

Token 是模型处理文本的基本单位，大约 1 个英文单词 = 1 个 token，1 个中文字 = 1-2 个 token。

```
"Hello, world!" → 4 tokens
"你好，世界！" → 6 tokens
```

每次调用 API，你发送的所有内容（对话历史 + 工具结果 + 系统提示）都要计入 token 数。超过上限，API 就会报错。

---

## 第一道防线：Snip（剪切）

最轻量的处理方式：**删除中间的冗余消息，保留头部和尾部**。

```
原始消息序列：
[系统提示] [用户1] [AI1] [工具调用1] [工具结果1] [用户2] [AI2] ... [用户N] [AI N]

Snip 之后：
[系统提示] [用户1] [AI1] ... [用户N-5] [AI N-5] [用户N] [AI N]
                              ↑ 中间的被删掉了
```

适用场景：对话历史里有很多"过渡性"的消息，比如"好的，我来看看"、"找到了，继续"这类不重要的内容。

---

## 第二道防线：Microcompact（微压缩）

针对**单个超长工具结果**的压缩。

有时候一个工具调用会返回大量内容，比如：
- 读了一个 5000 行的文件
- 执行命令返回了几千行日志
- 搜索结果返回了几十个文件的内容

Microcompact 会把这些超长结果压缩成摘要：

```
原始工具结果（5000 行代码）：
function foo() { ... }
function bar() { ... }
// ... 4998 行 ...

Microcompact 之后：
[已压缩：文件包含 5000 行，主要包含 foo、bar 等函数定义，
 完整内容已缓存，如需查看请重新读取]
```

关键设计：**压缩后的摘要里包含"如何恢复原始内容"的指引**，模型需要时可以重新读取。

---

## 第三道防线：Auto Compact（自动压缩）

当 token 数接近上限时，用 AI 生成整个对话的摘要，替换掉原始历史。

```typescript
// src/services/compact/autoCompact.ts
export function getAutoCompactThreshold(model: string): number {
  const contextWindow = getContextWindowForModel(model)
  return contextWindow - 13000  // 留 13000 token 的缓冲
}

// 当 token 数超过阈值时触发
if (tokenCount >= autoCompactThreshold) {
  // 调用 AI 生成摘要
  const summary = await compactConversation(messages)
  // 用摘要替换原始历史
  messages = [summary, ...recentMessages]
}
```

**电路断路器**（防止无限重试）：

```typescript
const MAX_CONSECUTIVE_FAILURES = 3

if (tracking.consecutiveFailures >= MAX_CONSECUTIVE_FAILURES) {
  // 连续失败 3 次，停止重试
  // 避免在无法压缩的情况下无限循环
  return { wasCompacted: false }
}
```

这个细节很重要。如果压缩本身也失败了（比如摘要太长），不能无限重试，否则会陷入死循环。

---

## 第四道防线：Session Memory（会话记忆）

前三道防线都是在**当前会话内**处理。Session Memory 解决的是**跨会话**的问题。

**场景：** 你今天用 Claude Code 重构了一个模块，明天继续工作时，Claude Code 还记得昨天做了什么吗？

默认情况下不记得。每次新会话都是从零开始。

Session Memory 的解决方案：**在后台定期提取关键信息，写入文件，下次会话时注入**。

```typescript
// src/services/SessionMemory/sessionMemory.ts（简化）

// 触发条件：token 数 AND 工具调用次数都达到阈值
function shouldExtractMemory(messages): boolean {
  const hasMetTokenThreshold = currentTokenCount > MIN_TOKENS_BETWEEN_UPDATES
  const hasMetToolCallThreshold = toolCallsSince > MIN_TOOL_CALLS_BETWEEN_UPDATES

  return hasMetTokenThreshold && hasMetToolCallThreshold
}

// 提取过程：用一个独立的 AI Agent 在后台运行
async function extractSessionMemory(messages, toolUseContext) {
  // 创建隔离的子 Agent，不影响主对话
  await runForkedAgent({
    prompt: buildExtractionPrompt(currentMemory, messages),
    canUseTool: onlyAllowEditMemoryFile,  // 只允许编辑记忆文件
    querySource: 'session_memory',
  })
}
```

提取出来的内容保存在 `~/.claude/projects/<项目ID>/session-memory.md`，格式大概是：

```markdown
# 项目记忆

## 当前任务
正在重构用户认证模块，目标是将 JWT 验证逻辑从 middleware 中分离出来。

## 已完成
- 创建了 src/auth/jwtValidator.ts
- 更新了 src/middleware/auth.ts，移除了内联验证逻辑

## 待完成
- 更新单元测试
- 更新 API 文档

## 重要发现
- 项目使用 RS256 算法，不是 HS256
- 公钥存储在 config/jwt-public.pem
```

下次会话时，这个文件会被自动注入到上下文，Claude Code 就"记得"之前做了什么。

---

## 四层防线的触发顺序

```
对话进行中
  │
  ├─ token 数 < 70% → 正常运行
  │
  ├─ token 数 70-85% → Snip 开始工作
  │                    删除中间冗余消息
  │
  ├─ token 数 85-90% → Microcompact 开始工作
  │                    压缩超长工具结果
  │
  ├─ token 数 > 90% → Auto Compact 触发
  │                   AI 生成整体摘要
  │                   替换历史消息
  │
  └─ 每隔一段时间 → Session Memory 后台提取
                    写入持久化文件
```

---

## 一个有趣的细节：只读 Agent 不注入 git 状态

Claude Code 里有很多专门的子 Agent，比如 `Explore`（探索代码库）和 `Plan`（制定计划）。

这些 Agent 是只读的，不会修改文件。所以 Claude Code 在给它们构建上下文时，**故意不注入 git 状态**：

```typescript
// src/tools/AgentTool/runAgent.ts
const resolvedSystemContext =
  agentDefinition.agentType === 'Explore' ||
  agentDefinition.agentType === 'Plan'
    ? systemContextNoGit  // 去掉 git 状态
    : baseSystemContext   // 完整上下文
```

原因：git 状态可能有几千行（大型项目的 `git status` 输出），对只读 Agent 完全没用，白白浪费 token。

这个细节体现了一个重要原则：**根据 Agent 的职责裁剪上下文，不要一刀切地注入所有信息**。

---

## 小结

Claude Code 的上下文管理策略：

| 层次 | 触发时机 | 处理方式 |
|------|---------|---------|
| Snip | token 数较高 | 删除中间冗余消息 |
| Microcompact | 单个工具结果过长 | 压缩为摘要 |
| Auto Compact | token 数接近上限 | AI 生成整体摘要 |
| Session Memory | 定期触发 | 提取关键信息，跨会话持久化 |

每一层都有电路断路器，防止失败时无限重试。
