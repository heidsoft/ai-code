# Claude Code Agent 为什么好用？——设计秘密深度解析

读完源码，我找到了 7 个让 Claude Code 比其他 AI 工具好用的核心设计决策。

---

## 一、系统提示是精心工程化的产品，不是随便写的文字

大多数人以为系统提示就是"告诉 AI 你是谁"。Claude Code 的系统提示是一个**工程化的产品**，有严格的分层结构。

### 分层缓存架构

系统提示被分成四个区域，每个区域有不同的缓存策略：

```
[归因头部]          → cacheScope: null（不缓存，每次变）
[CLI 前缀标识]      → cacheScope: 'org'（组织级缓存）
[静态核心内容]      → cacheScope: 'global'（跨组织全局缓存！）
[动态内容]          → cacheScope: null（不缓存）
    ↑
SYSTEM_PROMPT_DYNAMIC_BOUNDARY 分隔线
```

静态核心内容（工具使用规则、代码风格指南、安全规范）可以被**所有用户共享缓存**，大幅降低 API 成本。

### 系统提示的内容设计

看 `src/constants/prompts.ts`，系统提示不是一段话，而是多个精心设计的章节：

```
# Doing tasks（任务执行规范）
  - 不要超出要求范围
  - 不要添加不必要的错误处理
  - 不要过早抽象
  - 先读文件再修改
  - 不要创建不必要的文件

# Executing actions with care（谨慎执行）
  - 可逆操作 vs 不可逆操作的区分
  - 影响共享系统的操作要先确认
  - 遇到障碍不要用破坏性操作绕过

# Using your tools（工具使用规范）
  - 优先用专用工具，不要用 Bash 替代
  - 独立操作并行执行
  - 依赖操作顺序执行

# Communicating with the user（沟通规范）
  - 第一次工具调用前简短说明要做什么
  - 发现关键信息时更新进度
  - 假设用户已经离开，给出完整上下文
```

**关键洞察：** 这些规则不是"让 AI 更聪明"，而是**纠正 AI 的默认行为偏差**。AI 天然倾向于：
- 做超出要求的事（"顺便优化一下"）
- 添加不必要的注释和文档
- 过度设计（"以后可能用到"）

系统提示的核心工作是**约束这些偏差**。

## 二、工具描述是给模型的"使用说明书"，极其精细

看 `src/tools/BashTool/prompt.ts`，BashTool 的描述不是"执行 bash 命令"，而是一份详细的操作手册：

```
# 工具偏好（优先用专用工具）
- 搜索文件：用 GlobTool，不要用 find 或 ls
- 搜索内容：用 GrepTool，不要用 grep 或 rg
- 读文件：用 FileReadTool，不要用 cat/head/tail
- 编辑文件：用 FileEditTool，不要用 sed/awk

# 多命令执行规则
- 独立命令：单条消息里并行发多个工具调用
- 依赖命令：用 && 链接在一个调用里
- 用 ; 分隔时：前面失败了后面还会执行

# git 操作规范
- 永远不要跳过 hooks（--no-verify）
- 永远不要 force push 到 main/master
- 提交失败时创建新 commit，不要 amend

# sleep 使用规范
- 不要在可以立刻运行的命令之间 sleep
- 后台任务完成会通知你，不要轮询
- 不要用 sleep 循环重试失败的命令
```

**关键洞察：** 工具描述解决的是**模型的坏习惯**。模型天然会：
- 用 `cat` 读文件而不是专用工具
- 用 `sleep 5` 等待而不是用通知机制
- 在 commit 失败时用 `--amend` 而不是新建 commit

每一条规则背后都是一个真实的 bug 或用户投诉。

---

## 三、并行工具调用是一等公民

系统提示里反复强调并行：

```
# Using your tools
- 你可以在单条响应里调用多个工具
- 如果多个工具调用之间没有依赖关系，全部并行调用
- 尽量最大化并行工具调用以提高效率
```

BashTool 的描述里也有：

```
# 多命令执行
- 如果命令独立且可以并行，在单条消息里发多个 BashTool 调用
- 例如：需要同时运行 "git status" 和 "git diff"，
  发一条消息包含两个并行的 BashTool 调用
```

**为什么重要：** 串行执行 3 个文件读取需要 300ms，并行只需要 100ms。对于需要读取大量文件的任务（代码审查、重构），这个差距会累积到几十秒。

## 四、上下文注入的精确控制

`src/utils/api.ts` 里的 `prependUserContext` 函数揭示了一个关键设计：

```typescript
// 用户上下文（git 状态、CLAUDE.md 等）通过 user message 注入
// 而不是放在 system prompt 里
return [
  createUserMessage({
    content: `<system-reminder>
As you answer the user's questions, you can use the following context:
# claudeMd
${claudeMd}
# currentDate
Today's date is ${date}.
# gitStatus
${gitStatus}

IMPORTANT: this context may or may not be relevant to your tasks.
You should not respond to this context unless it is highly relevant.
</system-reminder>`,
    isMeta: true,  // 标记为元数据，不显示给用户
  }),
  ...messages,
]
```

**为什么这样设计：**
1. 动态内容（git 状态、日期）放在 user message 里，不会破坏 system prompt 的缓存
2. `isMeta: true` 标记让 UI 不显示这条消息，但模型能看到
3. `<system-reminder>` 标签告诉模型这是系统注入的上下文，不是用户说的话

## 五、子 Agent 的系统提示设计

看 `src/tools/AgentTool/built-in/generalPurposeAgent.ts`，通用子 Agent 的系统提示只有几行：

```
你是 Claude Code 的一个 Agent。根据用户消息，使用可用工具完成任务。
完整完成任务——不要镀金，但也不要半途而废。

你的优势：
- 在大型代码库中搜索代码、配置和模式
- 分析多个文件以理解系统架构
- 调查需要探索多个文件的复杂问题

指南：
- 搜索文件时：不知道在哪里就广泛搜索
- 分析时：先宏观后微观
- 要彻底：检查多个位置，考虑不同命名约定
- 永远不要主动创建文档文件（*.md）或 README
```

**关键设计：** 子 Agent 的提示比主 Agent 简洁得多。原因：
1. 子 Agent 继承了父 Agent 的上下文（通过 Fork 机制）
2. 子 Agent 有明确的单一职责，不需要全套规则
3. 精简的提示 = 更少的 token = 更快的响应

## 六、错误处理的哲学：诊断优先，不要盲目重试

系统提示里有一段很有意思的规则：

```
如果一种方法失败了，在换策略之前先诊断原因——
读错误信息，检查你的假设，尝试有针对性的修复。
不要盲目重试相同的操作，但也不要在第一次失败后就放弃可行的方法。
只有在真正调查后仍然卡住时，才用 AskUserQuestion 工具升级给用户，
而不是把遇到阻力作为第一反应。
```

这解决了 AI 的两个常见问题：
1. **盲目重试**：同样的命令失败了，再试一次，再试一次...
2. **过早放弃**：遇到一个错误就说"我无法完成这个任务"

**正确的行为：** 读错误 → 理解原因 → 针对性修复 → 如果真的卡住了才问用户。

## 七、代码风格规则：对抗 AI 的"过度工程"倾向

系统提示里有大量针对代码风格的规则，每一条都是在对抗 AI 的默认行为：

```
# 不要做超出要求的事
- bug 修复不需要清理周围的代码
- 简单功能不需要额外的可配置性
- 不要给没有修改的代码添加文档字符串、注释或类型注解

# 不要添加不必要的防御性代码
- 不要为不可能发生的场景添加错误处理
- 信任内部代码和框架的保证
- 只在系统边界（用户输入、外部 API）验证

# 不要过早抽象
- 不要为一次性操作创建 helper 或工具函数
- 不要为假设的未来需求设计
- 三行相似的代码比过早的抽象更好

# 关于注释（内部版本更严格）
- 默认不写注释
- 只在 WHY 不明显时才写：隐藏的约束、微妙的不变量、特定 bug 的 workaround
- 不要解释代码做了什么（好的命名已经做到了）
- 不要引用当前任务、修复或调用者
```

**关键洞察：** 这些规则不是"让 AI 写更好的代码"，而是**让 AI 写更少的代码**。AI 天然倾向于展示能力，写更多、更复杂的代码。这些规则强制 AI 保持克制。

---

## 总结：Claude Code 好用的本质

不是因为模型更聪明，而是因为**系统提示工程做得极其精细**：

1. **纠正偏差**：每条规则都针对 AI 的一个具体坏习惯
2. **分层缓存**：静态内容全局缓存，动态内容不缓存，成本最优
3. **并行优先**：工具调用并行化，速度最优
4. **精确注入**：上下文通过 user message 注入，不破坏 system prompt 缓存
5. **子 Agent 精简**：子 Agent 提示极简，继承父 Agent 上下文
6. **诊断优先**：失败时先诊断，不盲目重试也不过早放弃
7. **克制原则**：强制 AI 写更少的代码，不过度工程

这些设计背后是大量的用户反馈和迭代。每一条规则都是一个真实的问题被修复后留下的痕迹。
